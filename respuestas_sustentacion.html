SQLAlchemy ORM vs SQL Raw: ¿Por qué eligieron SQLAlchemy ORM?

¿Por qué ORM en lugar de SQL Raw?

Elegimos SQLAlchemy ORM porque permite definir las tablas como clases Python tipadas en lugar de escribir sentencias SQL manuales. Esto ofrece tres ventajas fundamentales:

  • Tipado y autocompletado: Cada columna se define con Column(Integer), Column(String), etc. El IDE detecta errores de tipo antes de ejecutar.
  • Mantenibilidad: Agregar una columna nueva es agregar una línea en la clase; no hay que buscar y modificar múltiples queries SQL.
  • Portabilidad entre motores: Nuestra configuración usa sqlite:///./risklab.db. Para migrar a PostgreSQL solo cambiamos DATABASE_URL en config.py sin tocar ningún modelo ni query ORM.

SQLAlchemy Core vs SQLAlchemy ORM

Curva de aprendizajeMás cercana a SQLMás Pythónica, más fácil de mantener

En nuestro archivo backend/app/models/db_models.py, cada tabla se mapea a una clase que hereda de Base:

Inyección de sesión con Depends(get_db): ciclo de vida y generador

Patrón del generador con yield

En backend/app/database.py, get_db() es un generador (usa yield), no una función convencional:

def get_db():
    db = SessionLocal()
    try:
        yield db          # ← FastAPI inyecta 'db' al endpoint
    finally:
        db.close()        # ← Se ejecuta SIEMPRE al finalizar el request

¿Por qué yield y no return?

FastAPI usa el protocolo de dependency generators. El bloque try/yield/finally garantiza:

  • Antes del yield: Se crea la sesión.
  • En el yield: FastAPI "pausa" el generador y entrega db al endpoint para que trabaje.
  • Después del yield (en finally): Se cierra la sesión incluso si el endpoint lanza una excepción.

¿Qué pasaría si no cerraran la sesión?

⚠️ Connection leak: Cada request abriría una conexión que nunca se devuelve al pool. Con SQLite (límite de ~64 conexiones simultáneas) y check_same_thread=False, el servidor se bloquearía rápidamente con errores OperationalError: database is locked. En PostgreSQL, se agotaría el pool de conexiones (por defecto 5-20).

Ciclo de vida completo de una sesión

  1. Request HTTP llega a FastAPI → el framework resuelve Depends(get_db).
  2. Se ejecuta get_db(): crea SessionLocal() y entra al generador.
  3. yield db — FastAPI inyecta la sesión al parámetro del endpoint.
  4. El endpoint ejecuta queries: db.query(Asset).filter(...).
  5. El endpoint retorna su respuesta.
  6. FastAPI continúa el generador → finally: db.close().
  7. La conexión se devuelve al pool de SessionLocal.

Uso en un endpoint

@app.get("/precios/{ticker}")

    ticker: str,
    svc: RiskService = Depends(get_risk_service),

La segunda NO vuelve a llamar a Yahoo Finance. La primera llamada ya persistió los datos en SQLite, y como la diferencia de días es 0 (≤ 3), la segunda llamada lee directamente de la BD.

¿Por qué functools.lru_cache no es buena alternativa?

  • Volatilidad de memoria: lru_cache almacena en RAM del proceso. Si el servidor se reinicia (deploy, crash), todo el caché se pierde.
  • No persiste entre instancias: En producción con múltiples workers (Gunicorn, Render), cada worker tiene su propio lru_cache → datos duplicados y desperdicio de RAM.
  • Sin expiración inteligente: lru_cache no soporta TTL (Time-To-Live). No puede saber si los datos tienen 3 días de antigüedad.
  • DataFrames grandes en RAM: Cachear DataFrames con lru_cache consume RAM rápidamente. SQLite usa disco, que es prácticamente ilimitado para este caso.
Ventaja SQLite: Los datos persisten entre reinicios, se comparten entre workers, y la BD ocupa ~487 KB en disco (risklab.db).

EWMA vs GARCH(1,1): diferencias fundamentales

EWMA (Exponentially Weighted Moving Average)

EWMA modela la volatilidad como un promedio ponderado exponencial de los rendimientos pasados al cuadrado:

σ²ₜ = λ·σ²ₜ₋₁ + (1−λ)·r²ₜ₋₁

Donde λ es el factor de decaimiento (típicamente 0.94 según RiskMetrics). Aquí α + β = (1−λ) + λ = 1.

GARCH(1,1)

GARCH agrega un término constante ω:

σ²ₜ = ω + α·r²ₜ₋₁ + β·σ²ₜ₋₁

Donde la varianza incondicional de largo plazo es:

σ²∞ = ω / (1 − α − β)

¿Por qué EWMA no tiene varianza incondicional finita?

En EWMA, α + β = 1 (equivalente a IGARCH). Sustituyendo en la fórmula de varianza incondicional:

σ²∞ = ω / (1 − 1) = ω / 0 → indefinida (∞)

Esto significa que EWMA no tiene un nivel de referencia al cual la volatilidad "regrese". Los shocks de volatilidad persisten indefinidamente.

Reversión a la media en GARCH(1,1)

En GARCH(1,1), si α + β < 1 (condición de estacionariedad), la varianza incondicional σ²∞ = ω/(1−α−β) es finita. Tras un shock de volatilidad alta, la volatilidad condicional tiende a regresar a este nivel de equilibrio → mean reversion.

Implementación en el proyecto

En src/garch_models.py comparamos ARCH(1), GARCH(1,1) y EGARCH(1,1) usando la librería arch con distribución t-Student:

modelo = arch_model(datos, vol="Garch", p=1, q=1,
                    mean="Constant", dist="studentst")
resultado = modelo.fit(disp="off")

El mejor modelo se elige por el criterio de información AIC (menor es mejor).

Backtesting de Kupiec: hipótesis y aplicación

¿Qué es el test de Kupiec (Proportion of Failures)?

Es una prueba estadística para evaluar si un modelo de VaR produce la cantidad correcta de excedencias (días en los que la pérdida real supera el VaR estimado).

Hipótesis

  • H₀: La proporción de excepciones observada es igual a la esperada (el modelo es correcto).
  • H₁: La proporción difiere significativamente (el modelo es inadecuado).

Estadístico

LR_POF = −2·ln[(1−p)^(T−N)·p^N] + 2·ln[(1−N/T)^(T−N)·(N/T)^N] ~ χ²(1)

Donde p = 1 − nivel_confianza, N = excepciones observadas, T = observaciones totales.

Ejemplo numérico: VaR al 99% con 15 excedencias en 250 días

  • Excepciones esperadas: T × (1−p) = 250 × 0.01 = 2.5
  • Excepciones observadas: 15 (¡6x más que lo esperado!)
  • Resultado: Se rechaza H₀ (p-valor ≈ 0) → el modelo es inadecuado.
  • Interpretación: El modelo está subestimando el riesgo, ya que predice pérdidas menores a las que realmente ocurren.

Implementación en el proyecto

En src/returns.py, función test_kupiec():

def test_kupiec(excepciones, n_observaciones, nivel_confianza):
    p = 1 - nivel_confianza
    p_hat = excepciones / n_observaciones
    lr = -2 * (
        (n - x) * np.log(1 - p) + x * np.log(p) -
        (n - x) * np.log(1 - p_hat) - x * np.log(p_hat)
    )
    p_value = 1 - stats.chi2.cdf(lr, df=1)
    return {"valido": p_value > 0.05, ...}

Se invoca en calcular_var() de risk_service.py contando cuántas veces el retorno del portafolio fue menor que -VaR_histórico.

Markowitz y programación cuadrática con no negatividad

Formulación del QP

min wT Σ w
s.a.: wT μ = μ*, Σᵢ wᵢ = 1, wᵢ ≥ 0 ∀i

¿Por qué es QP convexo?

La función objetivo wTΣw es cuadrática en w, y Σ (la matriz de covarianzas) es semidefinida positiva por definición. Toda función cuadrática con matriz SDP es convexa. Las restricciones son lineales (igualdad) y lineales con desigualdad (wᵢ ≥ 0), que definen un conjunto convexo. Por lo tanto, el problema es un QP convexo con un único óptimo global.

¿Qué cambia con la restricción wᵢ ≥ 0?

  • Sin restricción: Se permiten pesos negativos (ventas en corto). La frontera eficiente es más amplia.
  • Con restricción wᵢ ≥ 0: Se prohíben las ventas en corto. Los activos con peor relación rendimiento/riesgo quedan con peso cero (el optimizador los excluye). La frontera eficiente se contrae porque el espacio factible es más pequeño.

Implementación en el proyecto

En src/markowitz.py, usamos scipy.optimize.minimize con método SLSQP:

restricciones = {"type": "eq", "fun": lambda w: np.sum(w) - 1}
limites = tuple((0, 1) for _ in range(n))  # ← wᵢ ≥ 0

resultado = minimize(volatilidad, pesos_iniciales,
                     method="SLSQP",
                     bounds=limites,         # ← no negatividad
                     constraints=restricciones)

bounds=(0,1) impone que cada peso esté entre 0 y 1 → no hay ventas en corto.

Tres métodos de VaR: supuestos y limitaciones

Comparación de los tres métodos

AspectoParamétricoHistóricoMontecarlo
SupuestoNormalidad (μ, σ)Ninguno (usa datos reales)Distribución asumida (Normal en nuestro caso)
Colas pesadas❌ Subestima (colas finas)✅ Captura eventos realesDepende de la distribución elegida
Costo computacionalMuy bajo (fórmula cerrada)Bajo (ordenar datos)Alto (10,000 simulaciones)
Mejor escenarioCarteras lineales, horizonte cortoDatos históricos representativosCarteras no lineales (opciones)

¿Por qué calculamos los tres?

Cada método tiene fortalezas y debilidades complementarias. Usarlos en conjunto permite:

  • Si el VaR paramétrico es menor que el histórico → evidencia de colas pesadas.
  • El Montecarlo sirve como "cross-check" y permite modelar distribuciones más flexibles.
  • El backtesting de Kupiec valida empíricamente cuál modelo predice mejor.

¿Por qué CVaR complementa al VaR?

El VaR solo dice "con 95% de confianza, la pérdida no excederá X%", pero no dice nada sobre qué tan grande puede ser la pérdida cuando SÍ excede el VaR. El CVaR (Expected Shortfall) responde esa pregunta:

CVaR = E[L | L > VaR] = Promedio de pérdidas que exceden el VaR

Además, el CVaR es una medida coherente de riesgo (cumple subaditividad), mientras que el VaR no lo es. Subaditividad garantiza que la diversificación no aumente el riesgo medido.

Implementación

En src/var_cvar.py, los tres métodos y el CVaR se calculan sobre los rendimientos ponderados del portafolio. El Montecarlo usa 10,000 simulaciones (configurable en config.py).

Alpha de Jensen y desempeño relativo

¿Qué mide el Alpha de Jensen?

α = R_p − [R_f + β_p × (R_m − R_f)]

Mide el exceso de rendimiento del portafolio por encima de lo que el CAPM predice para su nivel de riesgo (Beta).

  • α > 0: El gestor del portafolio generó rendimiento superior al esperado → gestión activa exitosa. El inversionista recibió más retorno del que corresponde a su exposición al riesgo sistemático.
  • α < 0: Rendimiento inferior al esperado por CAPM → destrucción de valor.
  • α ≈ 0: Rendimiento coherente con el riesgo asumido (gestión pasiva).

Significancia estadística del Alpha

Para verificar si α es estadísticamente significativo, se examina el p-valor de la regresión OLS que calcula Beta. En src/capm.py:

slope, intercept, r_value, p_value, std_err = stats.linregress(
    datos["indice"], datos["activo"]
)
# intercept = alpha diario, p_value indica significancia

Si el p-valor del intercepto es < 0.05, el alpha es estadísticamente significativo al 95%.

Tracking Error e Information Ratio

Implementados en src/macro_benchmark.py:

  • Tracking Error: TE = σ(R_p − R_m) × √252 — Mide la volatilidad de la diferencia de rendimientos vs benchmark. Un TE alto indica que el portafolio se desvía mucho del benchmark (gestión activa).
  • Information Ratio: IR = (R_p − R_m) / TE — Normaliza el exceso de rendimiento por el riesgo activo (TE). Un IR > 0.5 sugiere una gestión activa exitosa sostenida.

Parámetros del modelo de Nelson-Siegel

Ecuación del modelo

y(τ) = β₀ + β₁ · [(1 − e^(−τ/λ)) / (τ/λ)] + β₂ · [(1 − e^(−τ/λ)) / (τ/λ) − e^(−τ/λ)]

Interpretación económica de los 4 parámetros

ParámetroInterpretaciónComportamiento
β₀Nivel de largo plazoAsíntota horizontal de la curva cuando τ → ∞. Representa la tasa de interés de equilibrio a largo plazo.
β₁Pendiente (corto plazo)Diferencia entre tasas de corto y largo plazo. β₁ < 0 → curva normal (tasas suben con plazo). β₁ > 0 → curva invertida.
β₂Curvatura (joroba)Controla el "humped shape" a plazos medios. β₂ > 0 → joroba positiva, β₂ < 0 → valle.
λ (tau)Velocidad de decaimientoDetermina a qué plazo alcanza su máximo la curvatura. λ pequeño → el efecto de β₁ y β₂ se concentra en plazos cortos.

Método de estimación

En backend/app/services/fixed_income.py usamos Mínimos Cuadrados No Lineales vía scipy.optimize.curve_fit:

popt, pcov = curve_fit(ecuacion_nelson_siegel, m, y,
                       p0=[b0_init, b1_init, b2_init, tau_init],
                       maxfev=10000)

curve_fit minimiza la suma de cuadrados de los residuos entre las tasas reales del Tesoro (obtenidas de FRED) y la curva teórica de Nelson-Siegel. Los valores iniciales se estiman heurísticamente: β₀ ≈ tasa a 30 años, β₁ ≈ tasa corto plazo − tasa largo plazo.

Duración, Convexidad y Sensibilidad del precio de un bono

Definiciones

  • Duración Modificada: D* = D / (1 + y) — Mide la sensibilidad porcentual del precio ante un cambio de 1 punto porcentual en la tasa de rendimiento. D es la Duración de Macaulay.
  • Convexidad: Mide la curvatura de la relación precio-tasa. Es la segunda derivada del precio respecto a la tasa, normalizada por el precio.

Ejemplo: D* = 7, tasa sube 100 pb (+1%)

ΔP/P ≈ −D* × Δy = −7 × 0.01 = −7%

El precio del bono caería aproximadamente un 7%.

Comparación numérica ante shock de +200 pb

Supongamos D* = 7 y Convexidad C = 60:

MétodoFórmulaΔP/P
(a) Solo duración−D* × Δy = −7 × 0.02−14.00%
(b) Duración + Convexidad−D*×Δy + ½×C×(Δy)² = −14% + ½×60×0.0004−12.80%
(c) Reprice exactoDescontar flujos a y_nuevo≈ −12.6% (depende de los flujos)

¿Por qué la aproximación lineal subestima la caída?

La relación precio-tasa de un bono es convexa (cóncava hacia arriba), no lineal. La aproximación de solo duración asume una relación lineal (tangente en el punto actual). Para subidas grandes de tasa:

  • La línea tangente predice una caída mayor que la real (−14% vs −12.6%).
  • La convexidad corrige esto al agregar el término cuadrático positivo +½C(Δy)², acercándose al precio real.
Nota: Para subidas de tasa, la duración sobreestima la caída. Para bajadas de tasa, la duración subestima la subida del precio. En ambos casos, el efecto convexidad es beneficioso para el tenedor del bono.

Paridad put-call y verificación numérica

Enunciado de la paridad put-call

C − P = S − K·e^(−rT)

Donde C = precio del call, P = precio del put, S = precio del subyacente, K = strike, r = tasa libre de riesgo, T = tiempo al vencimiento.

¿Por qué debe cumplirse exactamente?

El principio de no arbitraje garantiza esta relación. Si C − P ≠ S − Ke^(−rT), un arbitrajista podría construir un portafolio con ganancia segura sin riesgo:

  • Si C − P > S − Ke^(−rT): Vender call, comprar put, comprar acción, pedir prestado Ke^(−rT) → ganancia sin riesgo.
  • Si C − P < S − Ke^(−rT): Operación inversa.

En un mercado eficiente, estos arbitrajes desaparecen instantáneamente, forzando la igualdad (salvo errores numéricos de punto flotante).

Verificación numérica

Con nuestro módulo de Black-Scholes en backend/app/services/derivatives.py, supongamos S=100, K=100, T=1, r=5%, σ=20%:

# Call
d1 = (ln(100/100) + (0.05 + 0.5×0.04)×1) / (0.20×1) = 0.35
d2 = 0.35 - 0.20 = 0.15
C = 100×N(0.35) − 100×e^(−0.05)×N(0.15) ≈ 10.45

# Put  
P = 100×e^(−0.05)×N(−0.15) − 100×N(−0.35) ≈ 5.57

# Verificación:
C − P = 10.45 − 5.57 = 4.88
S − K·e^(−rT) = 100 − 100×e^(−0.05) = 100 − 95.12 = 4.88 ✅

La paridad se cumple exactamente (salvo redondeo de punto flotante).

Concepto fundamental

El concepto financiero que garantiza esta relación es el no arbitraje (no-arbitrage principle): no puede existir una estrategia que genere ganancia positiva segura sin inversión neta y sin riesgo. Es el pilar fundamental de la valuación de derivados.

Vega y su conexión con stress testing

Definición de Vega

ν = ∂V/∂σ = S · N'(d₁) · √T

Donde N'(d₁) = norm.pdf(d₁) es la función de densidad normal estándar.

¿Por qué Vega es siempre positiva?

Porque N'(d₁) > 0 siempre (la PDF normal es siempre positiva), S > 0 y √T > 0. Esto aplica tanto para calls como para puts: mayor volatilidad siempre aumenta el valor de una opción, ya que aumenta la probabilidad de terminar deep in-the-money.

Implementación en el proyecto

En backend/app/services/derivatives.py:

# Gamma y Vega son iguales para call y put
gamma = norm.pdf(d1) / (S * sigma * np.sqrt(T))
vega = S * norm.pdf(d1) * np.sqrt(T)
# Se reporta vega/100 (cambio por 1% en volatilidad)

Si la volatilidad se duplica (σ → 2σ)

Para un call at-the-money (S = K), la Vega es máxima. Si σ pasa de 20% a 40% (Δσ = 0.20):

  • Aproximación lineal: ΔV ≈ ν × Δσ. Ejemplo: si ν = 40 (por 1%), ΔV ≈ 40 × 20 = +800% → claramente exagerado.
  • Problema: La relación V(σ) no es lineal para shocks grandes. Vega misma cambia con σ (la derivada de Vega respecto a σ se llama Vomma o Volga).

¿Por qué conviene repricing completo para shocks grandes?

Para shocks grandes (σ → 2σ), las griegas son aproximaciones de primer orden. La aproximación lineal ΔV ≈ ν·Δσ ignora los términos de orden superior (convexidad de la relación precio-volatilidad). El repricing completo con Black-Scholes recalcula el precio exacto evaluando la fórmula completa con el nuevo σ, capturando toda la no-linealidad.

Se toma el portafolio óptimo de Markowitz (pesos w*) y se aplica un shock de mercado hipotético (ej: −20%). La propagación a cada activo individual se realiza mediante el Beta del CAPM:

ΔRᵢ = βᵢ × shock_mkt

Patrón Singleton para el modelo ML

Implementación del Singleton

En backend/app/services/ml_service.py, la clase MLService implementa Singleton mediante __new__:

class MLService:
    _instance: Optional['MLService'] = None
    _model: Any = None

    def __new__(cls):
        """Solo crea UNA instancia."""
        if cls._instance is None:
            cls._instance = super(MLService, cls).__new__(cls)
            cls._instance._initialized = False
        return cls._instance

    def __init__(self):
        if self._initialized:
            return              # ← NO re-carga el modelo
        self._load_model()      # ← Carga desde disco UNA VEZ
        self._initialized = True

Si se hacen 100 llamadas a /predict:

El modelo se carga desde disco EXACTAMENTE 1 vez. Las 99 llamadas restantes reciben la misma instancia que ya tiene el modelo en memoria.

Diferencia entre __new__ y __init__

MétodoCuándo se ejecutaPropósito
__new__(cls)Antes de crear el objetoControla la creación de la instancia. Retorna la instancia (nueva o existente).
__init__(self)Después de crear el objetoInicializa la instancia ya creada. Se ejecuta cada vez que se llama al constructor.

Para Singleton, __new__ verifica si _instance ya existe. Si sí, retorna la existente sin crear una nueva. El flag _initialized evita que __init__ recargue el modelo en llamadas posteriores.

Problema analítico elegido

Predicción de señales de trading (clasificación multiclase: Buy / Hold / Sell). Dado el estado actual de un activo (indicadores técnicos), predecir si en los próximos 5 días el precio subirá >1.5%, bajará >1.5%, o se mantendrá estable.

¿Por qué shuffle=False?

En backend/train_model.py, el split es temporal:

split = int(len(X_train_final) * 0.8)
X_train, X_test = X_train_final.iloc[:split], X_train_final.iloc[split:]
# Equivalente a train_test_split con shuffle=False

Leakage temporal en series financieras

Si usáramos shuffle=True:

  • Datos del futuro (2025) aparecerían en el conjunto de entrenamiento.
  • El modelo aprendería patrones de datos que cronológicamente son posteriores a algunos datos del conjunto de test.
  • Esto infla artificialmente la accuracy porque el modelo "ya vio" el futuro.
  • En producción, el modelo fallaría porque nunca tendrá acceso a datos futuros.
Leakage temporal = utilizar información del futuro para predecir el pasado. Es el error más grave en ML aplicado a finanzas porque produce métricas ilusorias que no se reproducen en trading real.

Dockerfile multi-stage: fundamento y beneficios

Estructura del Dockerfile

Nuestro backend/Dockerfile usa python:3.11-slim como imagen base ligera:

FROM python:3.11-slim

# Dependencias del sistema (build-essential para compilar arch/GARCH)
RUN apt-get update && apt-get install -y --no-install-recommends \
    build-essential && rm -rf /var/lib/apt/lists/*

# Capa de dependencias Python (se cachea si requirements.txt no cambia)
COPY requirements.txt .
RUN pip install --no-cache-dir -r requirements.txt

# Código fuente
COPY . .

CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]

¿Qué se instala en la etapa builder que NO pasa al runtime?

En un esquema multi-stage ideal, la primera etapa (builder) instala build-essential (gcc, make, headers C) necesarios para compilar extensiones nativas (arch, numpy, scipy). La segunda etapa (runtime) solo copia los wheels compilados sin las herramientas de compilación.

En nuestro caso actual, usamos una imagen slim con build-essential integrado pero limpiamos las listas de apt para reducir tamaño.

Comparación de tamaño

EnfoqueTamaño estimado
Single-stage con python:3.11 (full)~600–900 MB
Multi-stage con python:3.11-slim~150–200 MB

Impacto en cold start (Render free-tier)

Render free-tier pone los servicios en "sleep" tras 15 minutos de inactividad. El cold start requiere descargar e iniciar la imagen Docker. Una imagen de 600 MB tarda ~30–60 segundos, mientras que una de 150 MB tarda ~10–20 segundos. La diferencia es significativa para la experiencia del usuario.

Truco de caché de capas

Se copia requirements.txt antes del código fuente. Si solo cambia el código (pero no las dependencias), Docker reutiliza la capa de pip install cacheada → builds 10x más rápidos en iteraciones de desarrollo.

Tests con pytest: override de Depends(get_db) y BD en memoria

Mecanismo de override para tests

En un conftest.py de pytest, se reemplaza la dependencia get_db por una que usa SQLite en memoria:

# conftest.py (esquema estándar de FastAPI testing)
from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker
from app.database import Base
from app.main import app

# BD en memoria — completamente aislada de producción
SQLALCHEMY_TEST_URL = "sqlite:///:memory:"
engine_test = create_engine(SQLALCHEMY_TEST_URL,
    connect_args={"check_same_thread": False})
TestSessionLocal = sessionmaker(bind=engine_test)

def override_get_db():
    db = TestSessionLocal()
    try:
        yield db
    finally:
        db.close()

# CLAVE: se reemplaza la dependencia real por el mock
app.dependency_overrides[get_db] = override_get_db

# Fixture que crea/destruye las tablas para cada test
@pytest.fixture(autouse=True)
def setup_db():
    Base.metadata.create_all(bind=engine_test)
    yield
    Base.metadata.drop_all(bind=engine_test)

¿Cómo funciona dependency_overrides?

FastAPI mantiene un diccionario app.dependency_overrides. Cuando una ruta declara Depends(get_db), FastAPI busca si existe un override para esa función. Si existe, usa el override en lugar de la función original → los tests nunca tocan la BD de producción.

¿Por qué cada test debe ser aislado?

  • Reproducibilidad: Si un test inserta datos que otro test lee, el orden de ejecución afecta los resultados.
  • Independencia: Cada test crea sus tablas en :memory: y las destruye al final → estado limpio garantizado.
  • Paralelización: Tests aislados pueden ejecutarse en paralelo sin conflictos.

Cinco tests obligatorios del proyecto (ejemplos)

  1. Test de health check: GET /health retorna status 200 y {"status": "ok"}.
  2. Test de validación 422: POST /var con pesos que no suman 1.0 → retorna 422 con errores descriptivos.
  3. Test de precios: GET /precios/AAPL retorna status 200 con la estructura esperada (ticker, precios, retorno_total).
  4. Test de CAPM: GET /capm?tickers=AAPL retorna activos con Beta, Alpha de Jensen y clasificación.
  5. Test de frontera eficiente: POST /frontera-eficiente retorna portafolio de mínima varianza y máximo Sharpe con pesos que suman 1.0.

Síntesis y recomendación de inversión

Resumen ejecutivo para el inversionista

Integrando los resultados de los módulos del proyecto, la recomendación se estructura así:

1. Composición del portafolio óptimo (Markowitz — Módulo 6)

El portafolio de máximo Sharpe optimizado con restricción de no negatividad (wᵢ ≥ 0) concentra los pesos en activos con mejor relación rendimiento/riesgo. Los activos con alto Beta y baja contribución al Sharpe quedan con peso cercano a cero. La composición exacta (ej: 35% MSFT, 25% AAPL, 20% GOOG, 20% AMZN) maximiza el rendimiento ajustado por riesgo.

2. VaR y resultado del backtesting de Kupiec (Módulos 5 y 2)

El VaR al 99% indica la pérdida máxima diaria esperada con alta confianza. Si el backtesting de Kupiec valida el modelo (p-valor > 0.05), nuestras estimaciones de riesgo son confiables. Si el modelo se rechaza, debemos alertar al inversionista de que las pérdidas podrían ser mayores a lo estimado.

3. Pérdida proyectada bajo stress testing vía Beta (Módulos 4, 6 y 13)

Ante un escenario de caída del mercado del −20%:

  • Si el Beta del portafolio es ~0.95, la pérdida proyectada sería ≈ −19%.
  • Un portafolio con más activos defensivos (Beta < 1) sufriría menos.
  • Recomendación: Si el inversionista no tolera este nivel de pérdida, rebalancear hacia activos defensivos.

4. Señales técnicas actuales (RSI, MACD — Módulo 7)

Las señales automáticas del panel semáforo proporcionan contexto táctico de corto plazo. Si la mayoría de los activos muestran señales de COMPRA (RSI < 30, MACD cruzando al alza), podría ser un buen punto de entrada. Si hay señales de VENTA generalizadas, considerar reducir exposición.

5. Cobertura con opciones (Black-Scholes — Módulo 10)

Para proteger el portafolio contra caídas, se puede comprar puts protectivos:

  • El costo de la cobertura se estima con Black-Scholes (precio del put).
  • El Delta del put indica cuántos contratos comprar para cubrir cada $100 de exposición.
  • Si la volatilidad implícita es alta (Vega alta), los puts son caros → considerar collar strategies.

Recomendación final integrada

Al inversionista: Mantenga el portafolio optimizado de Markowitz con rebalanceo trimestral. El VaR al 99% y el backtesting de Kupiec confirman que el modelo de riesgo es fiable. Ante el escenario de stress (−20%), la pérdida esperada es manejable (≈ β_port × 20%). Considere puts protectivos si las señales técnicas alertan sobrecompra generalizada (RSI > 70). El Alpha de Jensen positivo indica que el portafolio ha generado valor por encima de lo esperado para su nivel de riesgo.
