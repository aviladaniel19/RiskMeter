📋 Visión General

¿Qué es RiskLab?

RiskLab es una plataforma web interactiva que analiza el riesgo de una cartera de acciones en tiempo real. Combina un motor matemático en Python (backend FastAPI) con un tablero visual (SPA en HTML/Chart.js) que cualquier usuario puede usar desde el navegador.

🌎 Analogía del día a día

Imagina que tienes dinero invertido en varias empresas (AAPL, MSFT, GOOGL…). RiskLab es como un copiloto financiero: te dice si esas acciones están caras o baratas, cuánto dinero podrías perder en un mal día, cuál es la mejor combinación de activos para no perder tanto, y te avisa automáticamente si una acción da señales de peligro.

🧮 Motor matemático

8 módulos de cálculo en Python que procesan datos reales de Yahoo Finance y FRED API: indicadores técnicos, volatilidad GARCH, VaR, optimización de Markowitz y más.

🖥 Dashboard interactivo

Tablero web de una sola página (SPA) con Chart.js. Gráficos interactivos, paneles de señales semáforo, frontera eficiente con colorbar de Sharpe y ajuste dinámico de pesos.

⚙️ API REST con FastAPI

10 endpoints asíncronos con validación Pydantic, inyección de dependencias, manejo de errores HTTP y documentación automática en /docs (Swagger UI).

🐳 Despliegue con Docker

La aplicación puede ejecutarse localmente o en contenedor Docker con un solo comando: docker-compose up --build. Incluye CI/CD con GitHub Actions.

Análisis técnico e indicadores

Herramientas que los traders usan para decidir cuándo comprar o vender una acción, basadas en el comportamiento histórico del precio.

🌎 En palabras simples

Es como revisar el historial médico de una persona antes de darle un diagnóstico. Los indicadores técnicos son los "signos vitales" de una acción: temperatura (RSI), presión (Bandas de Bollinger), ritmo cardíaco (MACD). Con ellos el sistema detecta si la acción está "enferma" (sobrevendida) o "demasiado excitada" (sobrecomprada).

📈

Indicadores implementados en src/indicators.py

Peso en la nota: 10%
  • ✓
    SMA (Media Móvil Simple): promedio del precio de los últimos N días. Si el precio sube por encima de la SMA, es señal alcista. El dashboard permite elegir SMA 20, 50 o 200 días.
  • ✓
    EMA (Media Móvil Exponencial): igual que la SMA pero da más peso a los días recientes, reacciona más rápido. Visible en el gráfico de precios junto con la SMA.
  • ✓
    RSI (Índice de Fuerza Relativa): valor entre 0 y 100. Por encima de 70 = sobrecompra (caro, posible caída). Por debajo de 30 = sobreventa (barato, posible rebote). Se grafica con líneas de referencia.
  • ✓
    MACD: diferencia entre dos medias exponenciales. Cuando la línea MACD cruza hacia arriba la línea de señal → compra. Hacia abajo → venta. El histograma muestra la fuerza de esa diferencia.
  • ✓
    Bandas de Bollinger: canal que se ensancha cuando hay mucha volatilidad. El precio tocando la banda inferior suele ser señal de rebote; tocando la superior, de corrección.
  • ✓
    Oscilador Estocástico (%K y %D): mide dónde está el precio actual relativo al rango de los últimos 14 días. Señales similares al RSI: por encima de 80 = sobrecompra, por debajo de 20 = sobreventa.
🔗 Endpoint correspondiente: GET /indicadores/{ticker} — retorna todos los indicadores como serie temporal JSON. El frontend los grafica con Chart.js con selectores dinámicos de período y tipo de media móvil. 
# src/indicators.py — ejemplo del cálculo de RSI
def rsi(precios: pd.Series, periodo: int = 14) -> pd.Series:
    """RSI: mide la velocidad y magnitud de los cambios de precio."""
    delta = precios.diff()
    gain  = delta.where(delta > 0, 0.0)
    loss  = -delta.where(delta < 0, 0.0)
    avg_gain = gain.ewm(com=periodo-1, min_periods=periodo).mean()
    avg_loss = loss.ewm(com=periodo-1, min_periods=periodo).mean()
    rs = avg_gain / avg_loss
    return 100 - (100 / (1 + rs))

Rendimientos y propiedades empíricas

Caracterizar estadísticamente cómo se comportan los precios de cada acción: qué tan volátiles son, si sus cambios siguen una distribución "normal" y qué patrones presentan.

🌎 En palabras simples

Si tienes una acción que un día sube 5% y al día siguiente baja 3%, el rendimiento es ese porcentaje de cambio diario. RiskLab calcula el promedio, la variabilidad y la "forma" de esa distribución de cambios. Es como medir si una ruta al trabajo tarda siempre 20 minutos o si a veces tarda 10 y a veces 45: la variabilidad importa.

📊

Implementado en src/returns.py

Peso en la nota: 6%
  • ✓
    Rendimientos simples y logarítmicos: r = (P_t - P_{t-1}) / P_{t-1} y r = ln(P_t / P_{t-1}). Los logarítmicos son más adecuados para modelos matemáticos.
  • ✓
    Estadísticas descriptivas: media diaria y anualizada, volatilidad, asimetría (¿la distribución se inclina hacia pérdidas o ganancias?) y curtosis (¿las colas son más gruesas que la normal?).
  • ✓
    Gráfico Q-Q: compara los cuantiles empíricos con los de una distribución normal teórica. Si los puntos se alejan de la diagonal, los rendimientos tienen "colas pesadas" (eventos extremos más frecuentes de lo esperado).
  • ✓
    Boxplot con Stripplot: visualización combinada que muestra la distribución de los rendimientos y cada punto de dato real. Implementado como SVG dinámico en el dashboard.
  • ✓
    Prueba Jarque-Bera: test estadístico formal que evalúa si los rendimientos siguen una distribución normal. En la práctica casi siempre la rechazan (colas pesadas).
  • ✓
    Prueba Shapiro-Wilk: alternativa a Jarque-Bera, más potente para muestras pequeñas. Se reportan estadístico y p-valor con interpretación en lenguaje claro.
  • ✓
    Hechos estilizados: texto interpretativo automático sobre colas pesadas, agrupamiento de volatilidad y efecto apalancamiento.

Modelos ARCH/GARCH de volatilidad

Los modelos clásicos asumen que la volatilidad es constante (como suponer que el tráfico en una ciudad es siempre igual). Los modelos GARCH capturan que en los mercados hay períodos de calma y períodos de tormenta.

🌎 En palabras simples

Imagina el nivel de ruido en una cafetería: hay días tranquilos y horas de punta donde todo el mundo grita. La volatilidad del mercado es igual: los días tranquilos se agrupan con días tranquilos, y las crisis se acumulan. Un modelo GARCH aprende ese patrón y predice qué tan "ruidoso" será el mercado mañana basándose en el ruido de hoy.

⚡

Implementado en src/garch_models.py

Peso en la nota: 10%
  • ✓
    Justificación de heterocedasticidad: Test ADF (¿es la serie estacionaria?), Test ARCH-LM de Engle (¿hay efectos ARCH?), Ljung-Box sobre rendimientos al cuadrado. Todos con valores p y decisión estadística.
  • ✓
    ARCH(1): la volatilidad de hoy depende solo del shock de ayer. Modelo base.
  • ✓
    GARCH(1,1): la volatilidad de hoy depende del shock de ayer Y de la volatilidad de ayer. Captura la "memoria" del mercado.
  • ✓
    EGARCH(1,1): igual que GARCH pero además captura el efecto apalancamiento — las caídas generan más volatilidad que las subidas de igual magnitud.
  • ✓
    Tabla comparativa AIC/BIC: los criterios de información seleccionan automáticamente el mejor modelo. Menor AIC = mejor ajuste sin sobreajustar.

  • ✓
    Pronóstico 30 días: estimación de la volatilidad esperada para los próximos 30 días de trading, graficada como serie temporal.
  • ★
    Resumen econométrico completo: tabla de coeficientes con p-valores, significancia estadística (***/**/*), Log-Likelihood, AIC y BIC del modelo seleccionado.

CAPM y riesgo sistemático

El modelo CAPM (Capital Asset Pricing Model) responde a una pregunta clave: ¿cuánto rendimiento adicional merece una acción por el riesgo que toma respecto al mercado?

🌎 En palabras simples

El Beta (β) mide cuán nerviosa es una acción comparada con el mercado. Si el mercado sube 1% y la acción sube 1.5%, su Beta es 1.5 (agresiva). Si el mercado cae 10% y la acción cae solo 6%, su Beta es 0.6 (defensiva). Un carro deportivo tiene "Beta alto" en las curvas; un camión de carga tiene "Beta bajo". El CAPM usa ese Beta para calcular qué rendimiento exige el mercado por ese nivel de riesgo.

🧮

Implementado en src/capm.py

Peso en la nota: 6%
  • ✓
    Beta por regresión OLS: se regresa el rendimiento del activo contra el del benchmark (S&P 500). La pendiente de esa regresión es el Beta. También se reporta R², Alpha de Jensen (¿ganó más de lo que prometía el CAPM?) y el p-valor de la regresión.
  • ✓
    Clasificación automática: β < 0.8 = Defensivo 🟢, 0.8–1.2 = Neutro ⚖️, 1.2–1.5 = Agresivo 🔥, β > 1.5 = Especulativo ⚡. Con descripción del perfil y tickers sugeridos.
  • ★
    Tasa libre de riesgo desde FRED API: el parámetro Rf se obtiene automáticamente desde el T-Bill 3 meses (serie DGS3MO) en tiempo real. No se ingresa manualmente.
  • ✓
    Security Market Line (SML): gráfico de dispersión que muestra a cada activo en el espacio (Beta, rendimiento esperado) con la línea teórica del CAPM. Activos por encima de la SML tienen Alpha positivo.
  • ✓
    Discusión de riesgo sistemático vs. no sistemático: texto interpretativo completo sobre diversificación y por qué solo el riesgo sistemático merece compensación.
# src/capm.py — fórmula CAPM
# E(R_i) = R_f + β_i × (E(R_m) - R_f)
def calcular_capm(beta: float, rf: float, rendimiento_mercado: float) -> dict:
    prima_riesgo       = rendimiento_mercado - rf        # Prima del mercado
    rendimiento_esperado = rf + beta * prima_riesgo      # CAPM
    clasificacion      = clasificar_beta(beta)          # Agresivo/Neutro/Defensivo
    return {"rendimiento_esperado": rendimiento_esperado, "clasificacion": clasificacion}

Valor en Riesgo (VaR) y CVaR

El VaR responde a la pregunta más importante en riesgo financiero: "¿cuánto dinero puedo perder en un mal día?"

🌎 En palabras simples

Imagina que tienes $100 millones en acciones. El VaR al 95% te dice: "Con un 95% de probabilidad, no perderás más de $2.3 millones en un solo día." El 5% restante de los días (los peores), podrías perder más. El CVaR (Expected Shortfall) complementa esto respondiendo: "En esos días malos del 5%, ¿cuánto perderé en promedio?" — siempre es peor que el VaR.

🛡

Implementado en src/var_cvar.py

Peso en la nota: 10%
  • ✓
    VaR Paramétrico: asume que los rendimientos son normales. VaR = μ - z × σ. Rápido de calcular pero menos preciso cuando hay colas pesadas.
  • ✓
    VaR Histórico: usa directamente el percentil empírico de los rendimientos pasados. Si el peor 5% de días históricos tuvo pérdidas de 2.5%, ese es el VaR. No asume ninguna distribución.
  • ✓
    VaR Montecarlo (10,000 simulaciones): genera 10,000 escenarios aleatorios de rendimientos futuros basados en los parámetros históricos. El percentil 5% de esas simulaciones es el VaR. El más robusto de los tres.
  • ✓
    CVaR / Expected Shortfall: promedio de las pérdidas que superan el VaR. Siempre mayor que el VaR y más conservador. Recomendado por Basilea III para regulación bancaria.
  • ✓
    Tabla comparativa de 3 métodos: VaR diario y anualizado para cada método, con interpretación en lenguaje claro.
  • ★
    Backtesting — Test de Kupiec: valida estadísticamente si el modelo VaR es preciso. Cuenta cuántas veces la pérdida real superó el VaR predicho (excepciones) y lo compara con lo esperado. Si el p-valor < 0.05, el modelo falla.
# src/var_cvar.py — VaR por simulación de Montecarlo
def var_montecarlo(rendimientos, nivel_confianza=0.95, n_simulaciones=10000):
    mu, sigma = rendimientos.mean(), rendimientos.std()
    simulados = np.random.normal(mu, sigma, n_simulaciones)  # 10k escenarios
    alpha     = 1 - nivel_confianza
    var_diario = -np.percentile(simulados, alpha * 100)        # Percentil 5%
    return {"var_diario": var_diario, "var_anual": var_diario * np.sqrt(252)}

Optimización de portafolio — Markowitz

Harry Markowitz ganó el Nobel de Economía por demostrar que la combinación correcta de activos puede reducir el riesgo sin sacrificar rendimiento. RiskLab implementa su teoría completa.

🌎 En palabras simples

No pongas todos los huevos en la misma canasta. Pero tampoco cualquier canasta sirve: si todas se mueven igual (alta correlación), no proteges nada. La Frontera Eficiente es la colección de todas las mejores combinaciones posibles: para cada nivel de riesgo, es la mezcla que da el mayor rendimiento. El Ratio de Sharpe mide cuánto rendimiento obtienes por cada unidad de riesgo asumido.

⚙️

Implementado en src/markowitz.py

Peso en la nota: 10%
  • ✓
    Matriz de correlación interactiva: tabla de calor (heatmap) que muestra cómo se correlacionan los activos entre sí. Azul = correlación positiva, rojo = negativa. Activos poco correlacionados se diversifican mejor.
  • ✓
    Simulación de 10,000 portafolios: se generan 10,000 combinaciones aleatorias de pesos (que suman 100%). Para cada una se calcula rendimiento, volatilidad y Ratio de Sharpe.
  • ✓
    Frontera Eficiente con colorbar de Sharpe: nube de puntos donde el color va de naranja (bajo Sharpe) a verde (alto Sharpe). La curva del borde superior izquierdo es la frontera eficiente.
  • ✓
    Portafolio de Mínima Varianza: la combinación de pesos que minimiza la volatilidad del portafolio, encontrado por optimización numérica (scipy.minimize con restricciones).
  • ✓
    Portafolio de Máximo Sharpe: la combinación que maximiza el rendimiento por unidad de riesgo (el más eficiente). Se muestra con composición detallada por activo.
  • ★
    Composición ajustable por el usuario: el panel de composición permite al usuario introducir pesos manualmente con barra de suma en tiempo real, y recalcular todas las métricas de riesgo al instante.

Señales y alertas automáticas

El sistema evalúa automáticamente 5 indicadores técnicos para cada activo y genera una señal global de COMPRA, VENTA o MANTENER con panel visual tipo semáforo.

✨ Módulo nuevo en esta versión: Operacionaliza los indicadores técnicos del Módulo 1 como señales de trading concretas. Requiere texto interpretativo automático en lenguaje simple.
🌎 En palabras simples

Es como un semáforo para cada acción: verde = los indicadores sugieren comprar, rojo = sugieren vender, gris = esperar. El sistema revisa automáticamente si el MACD cruzó, si el RSI está en zona extrema, si el precio toca las bandas de Bollinger, y combina todas esas señales en un veredicto final. Sin que el usuario tenga que interpretar cada gráfico manualmente.

⚡

Implementado en src/signals.py

Peso en la nota: 7%

    Cruce del MACD: si la línea MACD cruza por encima de la señal → COMPRA; por debajo → VENTA. Detecta el momento exacto del cruce comparando el valor actual con el anterior.
  • ✓
    RSI en zonas extremas: RSI > 70 (sobrecompra) → VENTA. RSI < 30 (sobreventa) → COMPRA. Con el valor numérico actual visible en el panel.
  • ✓

  • ✓

Renta fija: Curvas de rendimiento, Nelson-Siegel, duración y convexidad

Las curvas de rendimiento (yield curves) son la "columna vertebral" del precio de los bonos y la política monetaria. RiskLab descarga tasas reales desde FRED, ajusta una curva teórica y calcula métricas de sensibilidad.

🌎 En palabras simples

Un bono es un "pagaré": promete pagarte X% anual durante 10 años. Pero ese X% depende de cuánto tiempo esperes: 1 año quizás pague 3%, 10 años quizás 4.5%. La curva de rendimientos es la relación entre el plazo y la tasa. La duración mide cuánto sube o baja el precio del bono cuando las tasas suben 1% — es como la "sensibilidad" del bono a cambios de tasa de interés.

  • ✓

Algoritmo: Nelson-Siegel paso a paso

Entrada: Vector de vencimientos T = [0.25, 1, 2, 5, 10, 30] años, vector de tasas observadas y_obs = [3.4, 3.8, 4.2, 4.5, 4.7, 4.9] %

Fórmula Nelson-Siegel:

y(T) = β₀ + β₁·exp(−λT) + β₂·(λT)·exp(−λT)
Donde:
• β₀ = nivel asintótico (largo plazo)
• β₁ = factor corto plazo (β₀ + β₁ = tasa a corto plazo)
• β₂ = factor de curvatura (captura forma de joroba)
• λ = parámetro de decaimiento (típicamente 0.1-1.0)

Pasos:

  1. Inicializar: β₀ = promedio(y_obs), β₁ = y_obs[0] − β₀, β₂ = 0, λ = 0.5
  2. Optimizar: Algoritmo Levenberg-Marquardt para minimizar: SSE = Σ(y(Tᵢ) − y_obs[i])²
  3. Validar: RMSE = √(SSE / n). Si RMSE < 0.1 bp, convergencia exitosa
  4. Interpolar: Para cualquier vencimiento T nueva, calcular y(T) = β₀ + β₁·exp(−λT) + β₂·(λT)·exp(−λT)

# Input: tasas FRED observadas

Duración y Convexidad

1. Duración de Macaulay: Promedio ponderado del tiempo hasta recibir flujos

D = Σ(tᵢ · PV(CFᵢ)) / Price
Donde PV(CFᵢ) = flujo en tiempo tᵢ descontado

2. Duración Modificada (la que usamos):

D* = D / (1 + y) ≈ −(ΔPrice / Price) / (Δy)
Sensibilidad de precio a cambios de tasa

3. Convexidad: Curvatura de la relación precio-tasa

C = Σ(tᵢ·(tᵢ+1)·PV(CFᵢ)) / Price / (1+y)²
Ajuste de segundo orden en Taylor

Caso práctico: Bono 7Y, Cupón 4%, VN 100

# Flujos de efectivo: cupones anuales + principal
CFs = [4, 4, 4, 4, 4, 4, 104]  # años 1-7

# Tasa actual desde curva NS: 4.58%
y = 0.0458

# Precio actual
Price₀ = 4/(1.0458¹) + 4/(1.0458²) + ... + 104/(1.0458⁷) = 99.78

# Duración de Macaulay: ponderada por tiempo
D = (1×4/(1.0458¹) + 2×4/(1.0458²) + ... + 7×104/(1.0458⁷)) / 99.78 = 6.32 años

# Duración Modificada
D* = 6.32 / (1 + 0.0458) = 6.03

# Convexidad
C = 58.24  # cálculo similar, en años²

# SHOCK DE TASA: +100 pb (y pasa a 5.58%)
Δy = +0.01

# Estimado lineal (solo duración)
ΔPrice_lineal = −D* × Price × Δy = −6.03 × 99.78 × 0.01 = −6.01$
Price_lineal ≈ 99.78 − 6.01 = 93.77

# Estimado lineal + convexidad
ΔPrice_convex = ½ × C × (Δy)² × Price / 10000 = ½ × 58.24 × 0.01² × 99.78 = +0.29$
Price_convex ≈ 93.77 + 0.29 = 94.06

# Precio exacto (reprecio con CF descontados a 5.58%)
Price_exacto = 4/(1.0558¹) + ... + 104/(1.0558⁷) = 94.12

# Error: Lineal = 0.35$ (−0.37%), Convex = 0.06$ (−0.06%), Exacto = 0%
💡 Insight clave: Para shocks pequeños (< 100 pb), la duración es suficiente. Para shocks grandes (> 200 pb), la convexidad hace una diferencia material — hasta 1-2% del precio. Los bonos con menor convexidad pierden más cuando suben las tasas.

Opciones: Black-Scholes y Greeks

Las opciones son derivados que dan el derecho (no la obligación) de comprar o vender un activo a un precio fijo en el futuro. RiskLab valora opciones europeas sobre los activos del portafolio usando el modelo Black-Scholes y calcula las cinco "Greeks" que miden sensibilidades.

🌎 En palabras simples

Una opción de compra (call) es un "boleto de lotería": pagas una pequeña prima hoy, y si la acción sube mucho, ganas la diferencia. Si baja, pierdes solo la prima (nunca pierdes más). Las Greeks miden cómo cambia el precio de esa opción si: la acción sube 1% (Delta), si sube el doble que ayer (Gamma), si sube la volatilidad esperada (Vega), si pasa un día (Theta), si suben las tasas (Rho).

Fórmula Black-Scholes

CALL: C = S·Φ(d₁) − K·e^(−rT)·Φ(d₂)
PUT: P = K·e^(−rT)·Φ(−d₂) − S·Φ(−d₁)

Donde:
d₁ = [ln(S/K) + (r + σ²/2)·T] / (σ·√T)
d₂ = d₁ − σ·√T
Φ(·) = función de distribución normal estándar acumulativa

Interpretación intuitiva:

  • S·Φ(d₁): Valor presente esperado del subyacente si se ejerce (ponderado por probabilidad en escenario BS)
  • K·e^(−rT)·Φ(d₂): Valor presente del strike, ponderado por probabilidad de ejercicio
  • Diferencia: Prima que pagamos hoy por la opción

Las cinco Greeks

Delta (Δ)

Fórmula: Δ_call = Φ(d₁), Δ_put = Φ(d₁) − 1

Rango: Call: [0, 1], Put: [−1, 0]

Interpretación: Si el subyacente sube 1$, la opción sube ~Δ $. Si tenemos 100 calls con Δ=0.60, equivale a 60 acciones long.

Práctica: Delta = "apalancamiento" implícito de la opción

Gamma (Γ)

Fórmula: Γ = Φ'(d₁) / (S·σ·√T)

Rango: [0, ∞)

Interpretación: Velocidad de cambio del Delta. Si sube el subyacente, ΔΔ/ΔS ≈ Γ. Alto Gamma = Delta muy sensible = posición no lineal.

Práctica: Gamma = "riesgo de re-hedge". Los hedgers temen el Gamma alto.

Vega (ν)

Fórmula: ν = S·Φ'(d₁)·√T / 100

Rango: [0, ∞) (igual para call y put)

Interpretación: Si la volatilidad sube 1%, el precio de la opción sube ~ν·100 %. Las opciones largo son largas volatilidad; short son cortas volatilidad.

Práctica: Vega = apuesta sobre volatilidad futura

Theta (Θ)

Fórmula: Θ = −(S·Φ'(d₁)·σ) / (2·√T) − r·K·e^(−rT)·Φ(d₂)

Rango: Típicamente negativo para long calls/puts

Interpretación: Decay temporal. Si todo lo demás permanece igual, la opción pierde Θ $ por día. Short options ganan Theta.

Práctica: Theta = "sangría de tiempo". Los vendedores de opciones viven de Theta.

Rho (ρ)

Fórmula: ρ_call = K·T·e^(−rT)·Φ(d₂), ρ_put = −K·T·e^(−rT)·Φ(−d₂)

Rango: Varía según estructura

Interpretación: Si las tasas suben 1%, el precio de la opción cambia ~ρ %. Menos importante que Delta, Vega, Theta.

Práctica: Rho = riesgo de política monetaria. Importa en carteras de bonos + opciones.

Volatilidad Implícita (Newton-Raphson)

Problema: Observamos que el call cuesta 11.50$ en el mercado (no 10.55$). ¿Cuál es la volatilidad que "implica" ese precio?

Algoritmo Newton-Raphson:

# σ₀ = adivinanza inicial
σ₀ = 0.40  # Volatilidad inicial (menor a 45%)

# Iteración 1
C(σ₀=0.40) = 9.87$      # Precio BS con σ=0.40
vega(σ₀) = 0.295$    # Sensibilidad
error = 11.50 − 9.87 = 1.63$

σ₁ = σ₀ + error / vega = 0.40 + 1.63 / 0.295 = 0.40 + 0.0552 = 0.4552

# Iteración 2
C(σ₁=0.4552) = 11.48$
vega(σ₁) = 0.298$
error = 11.50 − 11.48 = 0.02$ # Convergencia!

σ₂ = 0.4552 + 0.02 / 0.298 = 0.4552 + 0.0000672 ≈ 0.4553

# Comparación con σ histórica (45%) → sobrevaloración leve en el mercado

Stress testing y análisis de escenarios

Backtesting (Kupiec en Módulo 5) valida el VaR contra historia observada. Stress testing va al futuro: aplica escenarios hipotéticos extremos (algunos sin precedente) para estimar pérdidas bajo "tormenta perfecta".

🌎 En palabras simples

VaR te dice: "En el peor 5% de días históricos, pierdes $2M". Pero la historia podría no ser suficiente si hoy el mundo es distinto. Stress testing pregunta: "¿Qué pasa si suben las tasas 200 pb y cae el mercado 30% todo a la vez? ¿Cuánto pierdo?" Es la pregunta que hace un banco central o una reguladora de riesgo (Basilea III lo exige).

ΔP_NVDA = −17 × 0.02 = −0.34  → P_new = 140 × (1 − 0.34) = 92.40
Pérdida_NVDA = (92.40 − 140) / 140 = −34.0%

# COIN (D_implied = 22 años, más sensible por alto crecimiento esperado)
ΔP_COIN = −22 × 0.02 = −0.44 → P_new = 95 × 0.56 = 53.20
Pérdida_COIN = −44.0%

# Promedio ponderado del portafolio
Pérdida_Portafolio = 0.30×(−0.34) + 0.15×(−0.44) + ...
                    ≈ −22.3%

Valor_Portafolio = 100,000 × (1 − 0.223) = $77,700
Pérdida_Absoluta = −$22,300 # vs VaR base de $3,247

Escenario 2: Caída de mercado −30% (crash)

Mecanismo: El rendimiento de cada activo responde según su Beta (exposición sistemática)

R_activo = α + β × R_mercado + ε
En stress: ε tiende a 0, correlaciones → 1, solo importa β 
# Escenario 2: Crash de −30%

# Cada activo cae según su Beta
Caída_NVDA = −1.45 × (−0.30) = −43.5%  → P_new = 140 × 0.565 = 79.10
Caída_COIN = −2.10 × (−0.30) = −63.0%  → P_new = 95 × 0.37  = 35.15
Caída_MSTR = −2.35 × (−0.30) = −70.5%  → P_new = 82 × 0.295 = 24.19
# ... etc para AMD, PLTR, HOOD

# Beta promedio ponderado del portafolio
β_port = 0.30×1.45 + 0.15×2.10 + 0.15×2.35 + 0.20×1.62 + 0.15×1.58 + 0.05×1.85
β_port = 0.435 + 0.315 + 0.3525 + 0.324 + 0.237 + 0.0925 = 1.758

# Caída del portafolio
Caída_Portafolio = −1.758 × (−0.30) = −52.7% (más que el mercado!)

Valor_Portafolio = 100,000 × (1 − 0.527) = $47,300
Pérdida_Absoluta = −$52,700 # Catastrófico vs VaR base

Escenario 3: Aumento de volatilidad (σ → σ × 2)

Mecanismo: Volatilidad mayor → VaR paramétrico sube → opciones suben de valor (Vega positivo)

VaR_new = VaR_base × (σ_new / σ_base)
Supuesto simplificador: VaR escala linealmente con volatilidad 
# Escenario 3: Volatilidad se duplica

# VaR paramétrico base (1 día, 95%)
VaR_base = 3247$

# Con volatilidad duplicada
VaR_stressed = 3247 × 2 = 6494$

# Cambio de precio de acciones (aproximado, sin cambio de nivel)
# Volatilidad no afecta directamente el precio, pero sí la probabilidad de extremos
# En mercados reales, spike de vol → pánico → caídas de precio (~5-10%)

Caída_nominal ≈ −5% a −10% (correlacionado con el volstress)

Pérdida_Portafolio ≈ −7% (tomando promedio)
Valor_Portafolio = 100,000 × 0.93 = $93,000
Pérdida_Absoluta = −$7,000

# Pero el riesgo (VaR) es mayor: el portafolio ahora puede perder hasta $6,494 en un día

Escenario Combinado: "Tormenta Perfecta"

Todos los shocks simultáneamente: Tasas +200pb + Caída −30% + Vol ×2

# Cada efecto se compone

# Paso 1: Shock de tasa → −22.3%
Precio_P1 = 77,700

# Paso 2: Crash de mercado (sobre el precio ya reducido por tasa)
# El crash es de −52.7% del mercado, pero ahora se aplica a precios más bajos
Precio_P2 = 77,700 × (1 − 0.527) = $36,600

# Paso 3: Volatility spike causa pánico adicional (−5% más)
Precio_P3 = 36,600 × 0.95 = $34,770

# Pérdida total
Pérdida_Total = (34,770 − 100,000) / 100,000 = −65.2%
Pérdida_Absoluta = −$65,230

# Comparación
VaR_95% = $3,247      # Peor día histórico (5% tail)
Stress_Combinado = $65,230 # Pérdida en tormenta perfecta
Ratio = 65,230 / 3,247 = 20x

# ⚠️ Stress testing revela 20× el riesgo del VaR en escenario combinado

Reverse Stress Testing

Pregunta: ¿Cuál es el shock mínimo que causa una pérdida del 20%?

Algoritmo (búsqueda binaria):

# Búsqueda binaria: encontrar shock_mercado tal que pérdida = 20%

crash_min = 0
crash_max = 0.50  # 50% de caída

for iteración in range(10):
    crash_mid = (crash_min + crash_max) / 2
    
    # Evaluar pérdida con este shock
    pérdida = β_port × crash_mid
    
    if pérdida < 0.20:
        crash_min = crash_mid
    else:
        crash_max = crash_mid

# Solución
crash_crítico = crash_min
β_port = 1.758
crash_crítico = 0.20 / 1.758 = 11.4%

# Resultado: Una caída del mercado de solo 11.4% causa pérdida de 20% en el portafolio
# Es decir, el mercado puede caer 11.4% y este portafolio pierde $20,000 (20%)
⚠️ Insight crítico: Este portafolio de alto beta (1.758) amplifica movimientos de mercado. Un crash del 11.4% en el S&P 500 causa 20% de pérdida aquí. Es doblemente riesgoso que el mercado agregado.

Validación con Pydantic v2

# backend/app/models.py — validación avanzada del portafolio
class PortafolioRequest(BaseModel):
    tickers: list[str] = Field(..., min_length=2, max_length=20)
    pesos:   list[float] = Field(..., description="Deben sumar 1.0")
    nivel_confianza: Annotated[float, Field(ge=0.90, le=0.99)] = 0.95
    periodo: str = Field(default="2y", pattern=r"^\d+[dmy]$")

    @field_validator("tickers")              # Semana 4 — validators
    @classmethod
    def normalizar_tickers(cls, v):
        return [t.strip().upper() for t in v]  # "aapl" → "AAPL"

    @field_validator("pesos")
    @classmethod
    def verificar_suma(cls, v):
        if not (0.999 <= sum(v) <= 1.001):  # Tolerancia punto flotante
            raise ValueError("Los pesos deben sumar 1.0")
        return v

    @model_validator(mode="after")          # Valida múltiples campos
    def verificar_longitud(self):
        if len(self.tickers) != len(self.pesos):
            raise ValueError("Tickers y pesos deben tener la misma longitud")
        return self

⚡ async/await (Semana 4)

Todos los endpoints de FastAPI usan async def. Las llamadas a Yahoo Finance y FRED son I/O-bound — async permite atender múltiples solicitudes sin bloquear el servidor.

🔒 Manejo de errores HTTP (Semana 2)

Errores 404 cuando el ticker no existe, 503 cuando Yahoo Finance o FRED fallan, y un manejador personalizado de errores 422 (validación Pydantic) con detalle de cada campo inválido.
